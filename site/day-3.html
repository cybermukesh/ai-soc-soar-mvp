Day 3

Wazuh + OpenSearch Ingestion Pipeline

Completed

Day 3 moved the product from static alert preview to a real Wazuh-first ingestion path: connector config, live health checks, OpenSearch alert fetch, normalization, persistence, triage handoff, and auditable sync runs.

Connector Operations Completed

  • Connector persistence for name, type, base URL, username, masked secret, enabled state, health status, latency, and last checked time.
  • Admin-only connector upsert API with URL, username, and password validation before enabling a connector.
  • Environment seeding action for Wazuh and OpenSearch connectors.
  • Connector setup summary endpoint to show readiness and missing required fields.
  • Connector audit events for create/update, seed, and health-check actions.
  • Connector health history table for repeated operational checks.

Live Wazuh/OpenSearch Work Completed

  • Wazuh API health probe authenticates and verifies manager status.
  • OpenSearch health probe checks cluster health and returns cluster status.
  • OpenSearch query reads from the Wazuh alert index pattern.
  • Wazuh hits are normalized into the internal alert schema: rule, asset, user, network, MITRE, severity, raw event.
  • Normalized alerts are persisted in the local runtime database.
  • Live alert summaries now correctly identify the source as opensearch:wazuh-alerts, not sample fixtures.

New Ingestion Workflow

  • POST /api/v1/ingestion/wazuh/sync fetches OpenSearch alerts, normalizes them, stores them, runs triage, and records an ingestion run.
  • GET /api/v1/ingestion/status exposes stored alert count, triage record count, last run, run history, and active live source.
  • Ingestion run records include status, detail, fetched count, stored count, triaged count, source, and timestamp.
  • Verified run: fetched 5 live Wazuh alerts, stored 5, triaged 5, and recorded ingestion run #1.

Frontend Work Completed

  • Live Wazuh Ingestion panel added to the executive dashboard.
  • Sync button added for analysts/admins to pull live Wazuh alerts on demand.
  • Stored alerts, triage records, last sync status, and live source are visible in the UI.
  • Connector page includes ingestion control and recent ingestion run history.
  • UI remains responsive with no horizontal overflow in the current browser viewport.

Evidence Endpoints

  • GET /api/v1/connectors/wazuh/health
  • GET /api/v1/connectors/opensearch/health
  • GET /api/v1/connectors/{name}/history
  • GET /alerts/wazuh/recent?limit=25
  • POST /api/v1/ingestion/wazuh/sync?limit=100&triage=true
  • GET /api/v1/ingestion/status
